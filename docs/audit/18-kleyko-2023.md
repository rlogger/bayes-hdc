## Summary (200 words)

Part II of the Kleyko et al. two-part survey catalogues the entire applications landscape of HDC/VSA as of 2023. Its outline is:

- **§2 Application areas.**
  - **§2.1 Deterministic behaviour with HDC/VSA** — automata & instructions (§2.1.1), data-structure transmission / communications (§2.1.2.1) and distributed orchestration (§2.1.2.2), string processing (§2.1.3), factorisation via resonator networks (§2.1.4).
  - **§2.2 Similarity estimation with HVs** — word/text context HVs (BEAGLE, Random Indexing, PSI; §2.2.1), biomedical-signal similarity (heart-rate, respiration, iEEG; §2.2.2), image-descriptor similarity (§2.2.3).
  - **§2.3 Classification** — feature-vector classification (§2.3.1) covering language ID (Table 3), text (Table 4), acoustic signals incl. speech (Table 5), faults (Table 6), automotive data (Table 7), behavioural signals incl. activity recognition and emotion (Table 8), EMG (Table 9), EEG (Table 10), CTG/DNA/SELDI-TOF/multi-modal biomedical (Tables 11–14); image classification (§2.3.2, Table 15); structured-data classification (§2.3.3, Table 16).
- **§3 Cognitive computing & architectures** — holistic transformations & "Dollar of Mexico" (§3.1.1), analogical reasoning (§3.1.2), cognitive modelling incl. sequence memory and Raven's Progressive Matrices (§3.1.3), computer vision and scene analysis (§3.1.4), and architectures (SPAUN, APNN, HTM, LIDA; §3.2).
- **§4 Discussion**, **§5 Conclusion**.

Most-cited application clusters (by table size) are **EMG gesture recognition (Table 9, ~12 studies), activity recognition (Table 8), language identification (Table 3), EEG/iEEG (Table 10), image classification (Table 15)**.

### Coverage breakdown vs. survey "top tier"

The five most-populated single-task application clusters in Kleyko Part II §2.3 are:
1. **EMG gesture recognition** (Table 9) → covered by `emg_gesture_recognition.py`.
2. **Behavioural / activity recognition** (Table 8) → covered by `activity_recognition.py`.
3. **EEG/iEEG seizure detection & motor-imagery BCI** (Table 10) → **NOT covered**.
4. **Image / character classification** (Table 15) → covered by `image_classification.py`.
5. **Language identification** (Table 3) → covered by `language_identification.py`.

Plus the foundational cognitive-computing themes:
- **Analogical reasoning / "Dollar of Mexico"** (§3.1.1.1) → covered by `kanerva_example.py`.
- **Sequence / working memory** (§3.1.3.2) → covered by `sequence_memory.py`.
- **Word / context similarity (BEAGLE / RI)** (§2.2.1) → covered in spirit by `song_matching.py` (bag-of-words bundle-as-similarity).

### What we *don't* cover and whether we should

- **EEG / iEEG seizure detection (Table 10).** A top-tier application cluster — it is the third-most-populated medical application in the survey after EMG and EEG-BCI. We have no example. **This is the most defensible candidate for an additional example** if the user wants to broaden coverage.
- **Resonator networks for factorisation** (§2.1.4). The repository does have `bayes_hdc/resonator.py` (the module exists), but there is no runnable example demonstrating it. Kleyko §2.1.4 is short but conceptually distinctive (Frady & Kent et al.). Worth considering as a future demo, especially given that the source module already exists.
- **Communications / over-the-air superposition** (§2.1.2.1). Niche.
- **Automotive / driving-context classification** (Table 7). Niche.
- **Robotics / sensorimotor control** (§2.1.1.2 controllers, §3.2 architectures). Niche for a software library; closer to Kleyko's "use of HDC in cognitive architectures".
- **Neuromorphic-hardware integrations.** Out of scope for a JAX library — this is hardware/co-design, not a software demo. Correctly omitted.

The current example set therefore covers **6 of the survey's 7 top-tier application clusters and both core cognitive-computing themes (analogy, sequence memory)**. The set is defensibly aligned with the survey's coverage breakdown.

## Trivial fixes proposed

These are concrete edits that should be made *if* the user accepts the substantive findings. They are listed for the user to apply (per audit instructions, this report does not edit code).

1. **`examples/README.md` framing claim — does not currently exist.** The audit instructions state that this file *claims* "the application set mirrors what HDC is most-applied to in the recent literature (Kleyko et al. 2022 survey)". We searched the file (and every other `*.md` / `*.rst` / `*.py` in the repository) and found **no such sentence anywhere**. The framing claim is not in the codebase as of the audited revision. There is therefore no year/date inconsistency to fix in `examples/README.md` — but if a future revision *adds* such framing, the canonical wording should be:

   > *"The application set mirrors what HDC is most-applied to in the recent literature; see Kleyko, Rachkovskij, Osipov & Rahimi (2023), 'A Survey on Hyperdimensional Computing aka Vector Symbolic Architectures, Part II', ACM Computing Surveys 55(9): Article 175."*

   **Year:** use **2023** (ACM CS journal publication; arXiv v3 also dates to 1 August 2023). Do not cite as "2022" — the original arXiv v1 was December 2021, the journal-aligned v3 is August 2023, and ACM CS volume 55(9) is the 2023 issue. Citing "2022" would be inconsistent with both the journal of record *and* the dated v3 PDF.

2. **`README.md` — no canonical-survey reference exists.** A `grep` for "Kleyko", "survey", "2023" in `README.md` returns nothing. If the user wants a single-sentence master reference (analogous to how `kanerva_example.py` cites Kanerva 2010 in its docstring), the natural location is in the "About" or "Architecture" section of `README.md`, with the citation form in fix 1.

3. **`DESIGN.md` — no survey reference exists.** Optional: add a one-line pointer in `DESIGN.md` §6 ("When to reach for this library") of the form *"For the broader application landscape of HDC/VSA, see Kleyko et al. (2023)."* — costs one citation, gives the reader a way out of the library into the literature.

4. **`BENCHMARKS.md` — no survey reference exists.** No fix needed; benchmarks in this file are quantitative (TorchHD comparison) and don't require a literature citation.

These fixes are all **additions of a missing canonical citation, not corrections of a wrong one**. The codebase does not currently mis-cite Kleyko — it does not cite Kleyko at all. (See Substantive Finding 1.)

## Substantive findings (for user review)

1. **The Kleyko Part II survey is *not currently cited* anywhere in the repository.** This is the single biggest finding of this audit. The audit-instructions framing claim ("examples/README.md claims the application set mirrors Kleyko et al.") does not actually exist as text in the file. Either (a) the framing claim was removed before this audit ran, (b) it was planned but never written, or (c) it lived in a sibling document. Since the user explicitly named this survey as "the reference our application catalogue should align with", and since our application catalogue *does* align with it (see Paper → Code map above), **the right action is to add a single canonical citation in `examples/README.md` and `README.md`**. This is a SUBSTANTIVE gap, not a date typo.

2. **The application set *is* genuinely well-aligned with Kleyko's coverage.** We are not in a situation where a framing claim overstates alignment. 6 of the 7 top-tier application clusters in §2.3 + both core cognitive-computing themes (§3.1.1 analogy, §3.1.3.2 sequence memory) + the foundational similarity primitive (§2.2.1 bundle-as-similarity, in `song_matching.py`) are present. The only top-tier cluster we miss is EEG/iEEG seizure detection (Table 10) — see fix candidate below.

3. **Considered gap: an EEG seizure-detection example would close the only top-tier omission.** If the user wants the strongest possible case that the example set covers all of Kleyko's top-cluster medical applications, an `examples/eeg_seizure_detection.py` along the lines of Burrello, Schindler, Benini & Rahimi 2018–2021 (refs [123], [200], [201], [202], [203], [204] in Kleyko Part II) would be the natural addition. The pipeline — sliding window of multi-channel EEG → local-binary-pattern symbolic encoding → bundle → centroid classifier — is a near-twin of `emg_gesture_recognition.py` and would re-use most of the encoding infrastructure. (The `bayes_hdc.datasets` module already has a synthetic-EEG-style structure pattern in `load_emg`, so the loader story is feasible.) **Recommendation: open a tracking issue for this; do not block on it**.

4. **Considered gap: a resonator-network factorisation example.** `bayes_hdc/resonator.py` exists but has no runnable example. Kleyko §2.1.4 is short but distinctive — it's the headline "deterministic-behaviour" application in the survey other than communications. A short demo (factor a composite HV `c = a ⊗ b` back into `(a, b)` via resonator iterations) would surface a capability that already lives in the repository but that no user can discover without reading source. Lower priority than #3 but cheaper.

5. **Year discipline going forward: pick "2023" everywhere.** Whenever this survey is cited in any future commit (in code comments, docstrings, README, DESIGN, BENCHMARKS, paper write-ups, citation graph maps, …), use *2023*. The arXiv preprint is *also* dated 2023 in its v3 form (1 Aug 2023), so there is no longer a version-skew problem — only the v1 (Dec 2021) and v2 timestamps are ≤ 2022, and they predate the camera-ready material. Citing "Kleyko et al. 2022 survey" was a reasonable shorthand while only the early arXiv preprint existed; in 2026 it is **wrong** and the journal-of-record year is the canonical answer.
